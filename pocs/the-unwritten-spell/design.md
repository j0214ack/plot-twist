# 體驗與魔法系統

## 玩家角色：沒有手的施法者

「沒有手」是玩法不變量，不只是外觀設定。角色可以移動、閃躲、改變視線、指向目標與說話，但不能直接：

- 拾取或攜帶物品；
- 普通攻擊；
- 推拉物體；
- 按按鈕、開門或拉動機關；
- 從背包使用道具。

因此，連「把鑰匙放進鎖孔」這種普通操作，也會成為一次讓玩家發明因果機制的機會。

## 旁註：沉默的活羽毛筆

旁註在故事與 UX 中承擔同一個角色：把玩家的非結構化意圖，翻譯成世界能夠執行的程式。

旁註可以：

- 用動作、墨線與聲音呈現正在書寫；
- 在結果被法力縮減時指出哪裡改變了；
- 解釋法術為何崩塌、終止或遭法典拒絕；
- 對意外的系統互動做一句簡短吐槽；
- 記住最近生成的房間、牆與法術，協助解析「剛才那道牆」等指涉。

旁註不能：

- 主動提供關卡答案；
- 每次施法都重述玩家的話或要求確認；
- 替玩家挑選最有利的魔法；
- 自己移動鑰匙、傷害守衛或改變世界；
- 把 AI 的推測包裝成法典真相。

### Technical spike 的進度提示

隊友第一次試玩時，不能假設他們已讀過關卡流程。Baked 關卡在玩家已完成一個明確里程碑後，可以用獨立於旁註的固定 HUD 提示下一個既定目標；這不是 AI 替玩家發明解法，也不在里程碑之前洩漏答案。

第一個例外是守衛倒下、唯一鑰匙掉落後：右上角持續顯示「現在施咒讓鑰匙去打開門」，直到門解鎖。它不能只用幾秒後消失的旁註呈現。

Technical spike 尚未提供自由指向／選取時，compiler 的 `focusedEntityId` 必須跟著 baked 關卡的當前阻礙交接，而不是永遠停在守衛：

- `FOCUS-1`：守衛存活時預設 focus 是守衛；守衛倒下且門仍鎖住時改為門；門解鎖後改為傳送門。這只是場景 context，不得覆蓋玩家在 utterance 中明確說出的 actor 或 target。

### Technical spike 的 HUD 可讀性

桌面 HUD 以 1440 × 900 viewport 為基準；更大的 viewport 依寬、高之中較小的比例一起放大，最多 1.45 倍。較小螢幕不縮到基準以下，繼續使用既有窄螢幕 media queries，避免文字因整體縮放而更難閱讀。

### Mobile presentation

手機不是把桌面 HUD 等比例縮小。它與桌面共用同一個遊戲世界與施法 pipeline，但只呈現橫向遊戲畫面、精簡的雙方 HP、左下 virtual joystick、右下按住詠唱的麥克風，以及必要的 transcript、旁註、next-step 與 victory feedback。文字 spell console、咒語範例、桌面 status cards 與鍵盤說明不出現在手機模式。

一般手機瀏覽器第一次進入時，先提示電腦體驗較完整；若仍用手機，建議加入主畫面取得 app-like fullscreen。已從主畫面以 fullscreen／standalone display mode 啟動時略過提示。直向手機以旋轉提示阻擋遊戲操作，完整判準見 [Decision 0008](decisions/0008-mobile-presentation-and-pwa-shell.md)。

### Startup loading presentation

網頁與從主畫面啟動的 PWA 使用同一個 startup loading screen。它是啟動狀態的遮罩，不是假裝知道下載百分比的進度條。

- `LOAD-1`：loading screen 由初始 HTML 提供，在遊戲 JavaScript 建立 UI 以前就能顯示；web 與 installed PWA 不分叉成兩套畫面。
- `LOAD-2`：只有 Demo session bootstrap 已結束（ready、需要 access code，或可見錯誤）且遊戲世界已成功 render 第一幀後，loading screen 才淡出。兩個條件不假設固定完成順序。
- `LOAD-3`：第一幀可以使用既有 primitive fallback；CC0 GLB 後續載入成功時再替換。單一素材失敗不能讓玩家永遠卡在 loading。
- `LOAD-4`：loading screen 可以用羽毛筆、墨跡與不定進度動畫表達仍在工作，但不顯示虛假百分比、模型推理或固定倒數。
- `LOAD-5`：loading 結束後才由 access gate、mobile browser notice 或 portrait gate 接手阻擋操作；loading 自己淡出後必須停止攔截 pointer，並對輔助技術標記為隱藏。

### 結果式回饋

旁註根據 runtime event 決定是否回應：

| 事件 | 建議表現 |
| --- | --- |
| 普通成功 | 不說話，只做動作或短音效 |
| 正常法力消耗 | 更新 Mana Bar |
| 法術被明顯縮減 | 一句短文字或語音 |
| 第一次遇到某種代價 | 解釋一次，之後不重複 |
| 出現意外連鎖反應 | 吐槽或指出新問題 |
| 法典拒絕法術 | 清楚說明拒絕的世界規則 |

例句：

- 牆只生成三面：「你要的是牢房。你剩下的墨，只夠寫三面牆。」
- 守衛很快打破牆：「你說了要牆，沒有告訴世界它該撐多久。」
- 火焰波及鑰匙：「火確實只燒在牆裡。鑰匙也在牆裡。」
- 永久新增房間：「那不是幻象。你真的在世界裡多寫了一個房間。」
- 直接宣告死亡：「那是一個結局，不是一條讓它發生的法則。」

## 詠唱與生成等待

玩家按住詠唱鍵說話，放開後直接提交，不進入確認對話。AI 生成、驗證與載入程式期間，遊戲保持運作：守衛繼續攻擊，玩家則以降低的移動速度閃躲。

Request-based STT 回傳完整 transcript 時，手機必須立刻顯示「你說：……」，並在 compiler 書寫期間持續可見。這是讓玩家核對辨識結果的 echo，不是確認步驟；它不能等待 spell generation 完成，也不把第一版改成 realtime partial transcription。

### Technical spike 的死亡保護

目前模型生成時間可能長於一輪戰鬥容錯，玩家若在第一個咒語完成前死亡，就無法驗證核心生成體驗。因此 playable spike 中玩家仍會受擊、HP 仍會下降，但最低停在 1，永遠保持可操作。這是明確的 playtest guardrail，不是最終平衡；等生成延遲與閃躲手感達標後再重議死亡、重生與失敗條件。

第一版不做節奏遊戲。戰鬥壓力本身就是詠唱時的 active waiting。

模型延遲不應被偽裝成精確進度。旁註可以用持續書寫、墨跡累積與不同階段的音效呈現「仍在工作」，但不顯示虛假的百分比或模型私有推理內容。

## 不清楚的詠唱

玩家只需要表達希望發生什麼，不需要事先理解「範圍、強度、持續時間、專注」等設計維度。

若玩家只說「把守衛關起來」，AI 與 runtime 可以補出一個合理但未必完美的版本，例如貼近守衛、標準材質、有限強度的 enclosure。未說清楚的部分會由世界替玩家做選擇；玩家透過結果與旁註的短評，學會下一次可以說「小一點，但更堅固」。

原則：

- 不故意曲解玩家意圖來製造笑話；
- 優先產生可逆、可觀察、能部分完成意圖的結果；
- 法力不足時可縮小、變弱、縮短或只完成一部分；
- 完全無法 grounding 的目標可以失敗或只生成無害效果；
- 直接影響受保護結果的說法由法典拒絕，而不是偷偷換一個意思。

## 一句話裡有多個 action

玩家不需要一次只講一個 intent。一次 utterance 可以包含一到多個 action，系統會把它編譯成一個 `SpellBundle`，再依 dependency 逐一具現 modules。

理解玩家的話時要區分：

- action：要求新增一段因果機制；
- reference：指出 action 要作用於哪個既有物件或 artifact；
- constraint：限制 action 的範圍、材質、條件或持續時間。

這些語意不是 module 數量公式。Module 的邊界取決於一段 runtime mechanism 是否需要獨立 lifecycle、資源代價、反制方式、跨 module 引用，或是否改變世界可讀取的 capability。「生成一道牆，然後在裡面放火」通常需要兩個可獨立存在、引用與解除的 modules；「用火焰構成一道牆」則可能是一個共同建立與清除的 compound module。完整判準見 [Decision 0007](decisions/0007-mechanic-module-boundary-and-locomotion.md)。

玩家不需逐招確認。旁註可以用墨行與「第 1／2 句」呈現執行狀態，但仍遵守不顯示私有推理、不提供虛假進度的原則。實作邊界見 [Decision 0002](decisions/0002-reference-harness-boundary.md)。

## 法典：可以創造原因，不能宣告結果

以下說法沒有可遊玩的因果，因此不能直接生效：

- 「守衛接下來會死。」
- 「守衛 HP 變成零。」
- 「門現在是打開的。」
- 「我已經過關了。」

旁註可以把句子劃掉，讓法典回應：「世界拒絕沒有因果的結局。」玩家若改說「每當火球碰到鏡牆，就反彈並傷害發射者」，系統便可以生成一個正常參與碰撞與傷害計算的機制。

「鑰匙開鎖」不是直接宣告結果：鑰匙是具有 `unlocker` affordance 的 actor，開鎖是它對門執行的合法 interaction goal。若鑰匙尚未接觸門，generated mechanic 應先讓它嘗試抵達有效距離；玩家多說的「飛、滾、滑行」可以形成獨立 locomotion mechanism，但不是讓 interaction 從失敗變成功的隱藏通關詞。路徑被 generated geometry 封死時，鑰匙必須碰撞、重新尋路並在 bounded timeout 後放棄，由旁註說明找不到能接觸目標的路。互動邊界見 [Decision 0006](decisions/0006-causal-interaction-navigation.md)，locomotion 與 module 判準見 [Decision 0007](decisions/0007-mechanic-module-boundary-and-locomotion.md)。

房間與通道可以被生成，因為它們屬於允許修改的世界空間；唯一鑰匙、守衛生命、門鎖狀態與通關旗標則是受保護狀態。Generated code 可以影響它們，但必須經過世界既有的 affordance。

## 允許玩家把遊戲「玩壞」

繞過預想戰鬥，或用「牆＋火」等組合擊敗守衛，不等於系統失敗。只要結果來自可觀察、可反制、可能失敗的因果鏈，就是 PoC 想展示的自由度。

真正的問題是某一個組合沒有成本、風險或條件，而且能穩定解決所有情境。長期可以透過詠唱暴露時間、專注、移動懲罰、法力、材質反應與敵人反應處理 dominant strategy；PoC 不需要先把這些全部平衡完。

## 法力與代價

施法前不提供精確估價或效果預覽。玩家只能看到目前 Mana；法術具現後，Mana Bar 才依實際結果變化。

| 結果類型 | 顯示與代價 |
| --- | --- |
| 一次性火焰、推力、投射物 | 消耗藍色 Mana，之後恢復 |
| 需要專注的持續效果 | 具現後綁定紫色 Mana；維持期間可能降低移動速度 |
| 永久房間、通道或世界改寫 | 具現後把部分上限變成黑色封印 |

玩家可以主動把限制說進詠唱，換取更有利的分配，例如「只要我站在原地，這道牆就不要倒下」。但系統不能要求新手先學會完整的法術規格；沒有說出的部分仍由世界補齊並以結果教學。

## 尚未定案

- 詠唱與專注各自降低多少移動速度。
- 是否只能同時維持一個專注法術。
- 被封印的 Mana 在本關、本輪或其他時點恢復。
- 法力不足時，各類效果優先縮減哪一個維度。
- 旁註使用固定台詞、生成台詞，或兩者混合。
